# Solo-Con ERP: Phase 1 Specification (Direct Ledger)

## 1. Project Vision
A streamlined, mobile-first double-entry ledger designed specifically for a single business owner. This phase skips multi-user "Voucher" approval workflows in favor of a **Direct Journal Entry** model to track construction projects, suppliers, and financial health with zero administrative overhead.

---

## 2. Technical Stack
*   **Frontend:** Flutter 3.x (Single codebase for Android + iOS).
*   **Database (Remote):** Supabase (PostgreSQL) for ACID-compliant financial integrity.
*   **Database (Local):** `sqflite` for local caching and offline-first data entry (critical for poor-signal construction sites).
*   **State Management:** Riverpod 2.x for reactive UI updates.
*   **Reporting:** `pdf` package for on-device PDF generation and export.

---

## 3. Database Schema (PostgreSQL)

### A. Entities (The "Who")
Entities are strictly separated to prevent mixing client receivables with vendor payables.
*   **`projects`**: ID, Name, Model Type (With-Material or Labour-Rate), Status (Active/Closed).
*   **`customers`**: ID, Name, Phone.
*   **`suppliers`**: ID, Name, Phone.

### B. Chart of Accounts (The "Where")
Accounts represent the categories where money resides or is spent. **The Bank is an account, not an entity.**
*   **Asset Accounts**: Cash, Bank (e.g., HBL, Meezan), Client Receivables.
*   **Liability Accounts**: Supplier Payables.
*   **Income Accounts**: Project Revenue.
*   **Expense Accounts**: Material Costs, Labour Costs.

### C. Unified Ledger (`journal_entries`)
This is the **only** table that stores financial values. Every single transaction must generate a Debit row and a Credit row linked by the same `transaction_id`.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | UUID | Primary key. |
| `transaction_id`| UUID | Shared ID to link the Debit and Credit row together. |
| `account_id` | TEXT | Link to the specific Account (e.g., 'Bank', 'Labour_Costs'). |
| `project_id` | UUID | Associated project (Mandatory for Expense/Income). |
| `supplier_id` | UUID | Linked supplier (Required for Payables/Material). |
| `customer_id` | UUID | Linked customer (Required for Receivables/Revenue). |
| `debit` | Numeric | Increase in Assets/Expenses or decrease in Liabilities. |
| `credit` | Numeric | Decrease in Assets/Expenses or increase in Liabilities. |
| `description` | TEXT | Manual note/memo for the transaction. |
| `created_at` | Timestamptz| Auto-generated timestamp. |

## 4. Transaction Logic (The Fast-Entry Engine)

The Flutter UI will abstract the complexity of double-entry accounting. The user interacts with a "Transaction Picker," and the backend repository handles the exact routing.

| User Action | Row 1 (Debit) | Row 2 (Credit) |
| :--- | :--- | :--- |
| **Material Buy (Credit)** | Material Costs (Linked to Project) | Supplier Payable (Linked to Supplier) |
| **Labour Payment (Cash)** | Labour Costs (Linked to Project) | Cash Account |
| **Supplier Pay (Bank)** | Supplier Payable (Linked to Supplier) | Bank Account |
| **Client Billing** | Client Receivable (Linked to Customer)| Project Revenue (Linked to Project)|
| **Receive Payment** | Bank or Cash Account | Client Receivable (Linked to Customer)|

---

## 5. Flutter Architecture: Offline-First Flow

Due to unreliable internet at construction sites, the app must never block the user from entering data. 

**The Sync Protocol:**
1.  **Local Write:** The `LedgerRepository` intercepts the transaction and immediately writes the Debit/Credit pair to the local `sqflite` database. 
2.  **UI Update:** Riverpod reads the local database and immediately updates the UI ledgers.
3.  **Background Sync:** A background worker detects un-synced rows and pushes the `transaction_id` payload to Supabase when an internet connection is established.

## 6. Financial Reporting (PDF Exports)

The app utilizes the `pdf` package to generate three core documents directly from the local SQLite data:

1.  **Income Statement (P&L)**:
    *   **Formula**: `Total Project Revenue - (Material Costs + Labour Costs)`.
    *   **Filters**: Must be filterable by `project_id` to determine individual site profitability.
2.  **Balance Sheet**:
    *   **Assets Side**: Current summed balance of (Cash + Bank + Receivables).
    *   **Liabilities & Equity Side**: Current summed balance of (Supplier Payables) + Owner's Equity.
    *   *System Check:* The app must throw a warning flag if Assets do not equal Liabilities + Equity.
3.  **Supplier Ledger (Project-Wise)**:
    *   A statement of account showing all purchases (Credits to Payable) and all payments (Debits to Payable) for a specific Supplier, resulting in a **Net Outstanding Balance**.

---

## 7. Critical Operational Rules
Because this system lacks administrative oversight and mandatory receipt photos, the operator **must** adhere to the following rules:
1.  **Daily Tally**: The app's `Cash` and `Bank` ledger balances must be reconciled against physical wallets and actual bank statements at the end of every day.
2.  **Correction Protocol**: Never execute SQL `DELETE` or `UPDATE` commands on existing ledger rows. Mistakes must be fixed by posting a "Reversing Journal Entry" to maintain a mathematically pure audit trail.
